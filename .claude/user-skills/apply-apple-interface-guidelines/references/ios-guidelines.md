# iOS guidelines

Prefer platform-standard navigation, controls, gestures, terminology, and system-provided feedback. Preserve expected Back behavior, sheets, alerts, context menus, and destructive-action confirmation.

Design for touch without assuming touch is the only input. Support VoiceOver, Switch Control, hardware keyboards where relevant, and system accessibility settings.

Respect safe areas, localization, privacy-sensitive data, permission timing, and interruption recovery. Ask for permission in context and explain the user benefit before the system prompt appears.

Custom appearance is acceptable when it preserves the standard control's meaning, state, feedback, and accessibility behavior. Custom interaction needs a concrete product reason.
