# /recall - 記憶の検索と呼び出し

memory-vector と memory-graph の両方を検索し、過去に保存した記憶やファクトを呼び出す。

## Usage

```
/recall <query>
```

Examples:
```
/recall iPod Gadgetの設定について
/recall 前にWiFiが壊れた件
/recall iAP通信のデバッグ結果
```

## Instructions

このスキルが呼び出されたら、以下の手順で記憶を検索する。

### 1. クエリの決定

- 引数が指定されている場合、そのままクエリとして使う
- 引数がない場合、ユーザーに何を思い出したいか質問する

### 2. 両方のメモリソースを並列検索する

以下の2つのツールを**同時に**呼び出す:

**memory-vector** (セマンティック検索):
- ツール: `mcp__memory-vector__search_memory`
- パラメータ:
  - `query`: ユーザーのクエリをそのまま使う（リポジトリ名プレフィックスは付けない）
  - `limit`: 5（通常の context-injector より多め）
- 類似度が低い結果（20%未満）は除外する

**memory-graph** (ファクト/関係性検索):
- ツール: `mcp__memory-graph__graphiti_search`
- パラメータ:
  - `query`: ユーザーのクエリをそのまま使う
  - `num_results`: 10
  - `group_ids`: 指定しない（全グループを横断検索）

### 3. 結果の統合と提示

検索結果を以下の形式で整理して提示する:

- **memory-vector の結果**: 類似度が高い順に、内容・ソース・類似度を表示
- **memory-graph の結果**: ファクトや関係性を箇条書きで表示
- 両方の結果に共通するテーマや矛盾がある場合は指摘する

### 4. フォローアップ

結果を提示した後:
- 結果が不十分な場合、別のキーワードでの再検索を提案する
- 関連しそうなキーワードがあれば、追加検索を提案する
- ユーザーが結果に基づいて作業を続けたい場合はそのまま支援する
- コミットハッシュやコーディングセッションの詳細が必要そうな場合、`/entire explain <commit>` での追加調査を提案する

## 検索のコツ

- 広い検索をしたい場合: 一般的なキーワードを使う
- 特定のプロジェクトに絞りたい場合: プロジェクト名やリポジトリ名を含める
- 人や技術の関係性を知りたい場合: memory-graph の結果を重視する
- 具体的な手順やログを探したい場合: memory-vector の結果を重視する
